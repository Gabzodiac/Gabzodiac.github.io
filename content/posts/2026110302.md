### 避坑指南：为什么别人接豆包会封号？

很多人照着网上的“野路子”教程接豆包，没两天就发现账号被封了。核心原因就三个，咱们一个个说清楚。

#### 坑一：API Key 直接硬编码在代码里

很多新手喜欢这样写：

javascript
const apiKey = 'sk-xxx'; // 直接写在代码里，然后上传到GitHub


这是最致命的操作。如果你把代码推送到公开的GitHub仓库，别人用爬虫扫描你的代码，马上就能扒走你的Key。别人拿着你的Key去调用API，产生的费用算在你的头上，而且如果别人拿着你的Key去做违规的事，服务商封的就是你的账号。

**解决方案：**
一定要用环境变量！创建一个 `.env` 文件：


YUNWU_API_KEY=your-real-api-key


然后在代码里读取：

javascript
import 'dotenv/config';
const apiKey = process.env.YUNWU_API_KEY;


这样你的Key就不会暴露在代码里了。

#### 坑二：请求频率过高，触发了服务商的限流

豆包或火山引擎对短时间内大量请求是有限流机制的。如果你在Node.js里写了一个for循环，无脑发送100个请求，分分钟把你封了。

**解决方案：**
使用[云雾ai聚合平台](https://www.yunwuai.cc/)的**并发控制**机制。云雾平台本身对并发没有限制，但你要在代码里做合理的节流。可以用 `p-limit` 库来控制并发数。

bash
npm install p-limit


javascript
const pLimit = require('p-limit');
const limit = pLimit(5); // 同时最多5个请求

const tasks = urls.map(url => {
    return limit(() => fetch(url));
});


#### 坑三：请求格式错误，返回了非标准错误

有时候你发了一个错误的请求（比如 `model` 名字写错了，或者 `messages` 格式不对），服务商会返回一个“错误”的响应。如果你不做处理，一直重试，就可能被判定为恶意请求。

**解决方案：**
一定要用 `try...catch` 包裹你的请求逻辑，并且仔细阅读错误信息。上面的代码示例里已经包含了优雅的错误处理。当收到 `status: 429`（请求过多）或 `status: 400`（参数错误）时，不要再盲目重试，去检查你的参数配置。

### 进阶提升：如何让豆包应用性能提升10倍？（非必须，但很爽）

如果你的项目对响应速度有要求，比如做实时聊天的应用，[云雾ai聚合平台](https://www.yunwuai.cc/)支持**流式输出（Streaming）**。

流式输出意味着豆包不需要等整段话生成完才一起返回，而是生成一个字就立刻推一个字符到前端。用户体验会好很多，感觉像真人在打字。

实现也很简单，在 `create` 函数里加上 `stream: true` 参数：

javascript
const stream = await client.chat.completions.create({
    model: MODEL_NAME,
    messages: messages,
    stream: true, // 开启流式输出
});

for await (const chunk of stream) {
    process.stdout.write(chunk.choices[0]?.delta?.content || '');
}


这样改完之后，你的豆包应用就从“哑巴模式”变成了“对话模式”，用户体验直接起飞。
