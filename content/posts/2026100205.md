## 三、Java接入示例：从零到一的完整代码（一次过别翻车）

Java调GPT-5.1的OpenAI接口，我强烈推荐用 `openai-java` 这个官方SDK库或者HTTP客户端直接发请求。我用的是 Maven 项目，加依赖：

<dependency>
  <groupId>com.theokanning.openai-gpt3-java</groupId>
  <artifactId>client</artifactId>
  <version>0.12.0</version>
</dependency>


**关键注意点**：这个库要用最新的稳定版本，0.12.0是实测适配GPT-5.1的。老版本可能不支持新版模型。

然后配置Client：

import com.theokanning.openai.OpenAiService;
import com.theokanning.openai.completion.chat.*;

OpenAiService service = new OpenAiService(
    "你从云雾拿到的API_KEY", // 不要用代码里的占位符，直接把字符串粘贴进去
    0 // 默认超时（秒），可以往后延到30秒
);


注意这里超时设置很重要：官方文档说 `0` 表示默认，但建议给到 `60` 秒以上。因为GPT-5.1有一次流式推理过程，国内网络有时延。我一开始设为 `30` 秒，联调时老是SocketTimeout，改成 `120` 秒以后才稳定下来。

**重要避坑点**：超时时间不要太短。如果生产环境并发高，用网络请求库的 `ConnectTimeout` 和 `ReadTimeout` 分别设置，Connect可以短一点（比如5秒），Read设置成60秒以上。

## 四、流式响应（SSE）接入：Java版一次正确实现

GPT-5.1的流式输出是开发者喜欢的地方。很多团队不擅长处理SSE，导致联调时出现乱码。我提供一个稳定版本：

java
ChatCompletionRequest request = ChatCompletionRequest.builder()
    .model("gpt-5.1")
    .messages(List.of(
        new ChatMessage(ChatMessageRole.USER.value(), "写一个Python快排算法。")
    ))
    .stream(true) // 开启流式
    .build();

// 使用Service的流式方法
service.streamChatCompletion(request)
    .doOnError(Throwable::printStackTrace)
    .blockingForEach(chunk -> {
        if (chunk.getChoices().size() > 0) {
            String content = chunk.getChoices().get(0).getMessage().getContent();
            if (content != null) {
                System.out.print(content);
            }
        }
    });


**避坑指南**：
1. **阻塞 vs 异步**：`.blockingForEach` 是阻塞的，适合测试。生产环境建议改用 `.subscribe()` 或者配合Reactor的 `Flux`。
2. **超时控制**：流式请求的ReadTimeout要设成 `0` （无限），或者设成 `300` 秒以上，避免长时间推理时断开。
3. **乱码处理**：国内有的代理中间件可能对UTF-8字符集做奇怪处理，但云雾是直连没这类问题，我用 `System.out.print` 没出乱码。

## 六、联调一次过的技巧总结

**联调阶段往往是最容易出幺蛾子的**，除了上面提到的格式、网络、模型名称问题，还要注意几个点：

- **请求头**：有些内部项目会加自定义请求头，云雾相容OpenAI标准，不必加额外信息。但如果加了Authentication也没事，但务必用Bearer令牌格式。
- **日志记录**：Java客户端可以启用 HTTP 请求日志。比如在Maven加 `logging-interceptor`，打印出请求/响应内容，能快速定位错误是格式错还是模型不支持。
- **连错时，排除项目本身**：先用CURL测试一次：
bash
curl https://www.yunwuai.cc/v1/chat/completions \
  -H "Content-Type: application/json" \
  -H "Authorization: Bearer YOUR_API_KEY" \
  -d '{
    "model": "gpt-5.1",
    "messages": [{"role": "user", "content": "Hello"}]
  }'

如果CURL能正常返回，说明Java代码的问题大概率在SDK版本/超时/序列化上。

- **错误码解读**：常见云雾返回的错误码及含义：
  - 400：请求格式有误（最常见）
  - 401：API Key错误或过期
  - 429：限流（但云雾不限制，未遇到）
  - 500/502：内部错误，重试即可

**联调现场一次过的真实经历**：我按上面写完Curl后Java代码跑通了，大概花了15分钟。对比以前接入官方OpenAI试了2小时才搞定，省了不少时间。
